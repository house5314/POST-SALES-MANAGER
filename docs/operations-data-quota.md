# 운영 주체·데이터 갱신·API 쿼터 (PoC 기준)

심사·감사에서 **“누가 운영하는지, 데이터가 얼마나 자주 바뀌는지, API 한도는 어떻게 보는지”**를 코드·문서로 대응하기 위한 정리입니다. 수치·한도의 **최종 확정 값**은 각 기관 포털·키 발급 계약서가 우선입니다.

## 1. 운영 주체 (역할 분리)

| 구분 | PoC(현재 저장소) | 본격 도입 시(권장) |
|------|------------------|---------------------|
| **애플리케이션** | 과제 PoC — 우정사업본부 AI 공모·내부 실증용 웹앱 | **도입 우정청(또는 본부 지정 부서)** 가 서비스 소유·배포·장애 대응 |
| **API 키·계약** | `.env.local` / 서버 환경 변수로 관리(개발자·과제 담당) | **정보기술·데이터 거버넌스** 담당이 키 발급·회전·모니터링 |
| **원천 데이터** | 소상공인시장진흥공단·국토교통부·행정안전부 등 **각 제공 기관** | 동일(원천의 정확성·갱신은 기관 책임) |
| **AI 모델** | `GOOGLE_GENERATIVE_AI_*` 선택 사용 — `docs/ai-internal-network.md` | 동일 + 내부망 게이트웨이 시 **보안·개인정보** 담당과 협의 |

**PoC 문구:** 본 저장소는 **운영 서비스가 아닌 실증용**이며, 상용화 시 운영 주체·SLA·민원 창구는 **도입 기관의 내부 규정**으로 확정합니다.

## 2. 데이터 “갱신”의 두 층

| 층 | 의미 | 본 앱 동작 |
|----|------|------------|
| **A. 원천 DB 갱신** | 소진공·국토부 등이 자체 DB를 얼마나 자주 갱신하는지 | **앱이 제어하지 않음.** 각 API·데이터셋 설명서·공지에 따름. |
| **B. 화면에 보이는 시점** | 사용자가 조회한 순간의 API 응답 | **요청 시마다** Route Handler가 공공 API를 호출(PoC는 **서버 측 캐시 없음**이 일반적). 즉 “실시간”은 **조회 시점 스냅샷**입니다. |

**권장 고지(창구·내부):** “지표·목록은 **조회한 순간**의 공공 API 결과이며, 어제와 오늘 수치가 다를 수 있습니다.”

## 3. API 쿼터·트래픽 (공공데이터포털 등)

| 항목 | 설명 |
|------|------|
| **한도의 근거** | `PUBLIC_DATA_API_KEY`로 호출하는 `apis.data.go.kr` 트래픽은 **공공데이터포털 이용약관·해당 API 상품(활용신청) 조건**을 따릅니다. 일일·월간 한도는 **키·기관·상품마다 상이**하므로 포털 “마이페이지” 및 API 문서의 **트래픽/제한** 안내를 확인해야 합니다. |
| **PoC에서의 호출 패턴** | 대표적으로 **행정동·업종 필터 변경 시** 상가 목록 재조회, **동 선택 시** 단지 목록 조회, **업체 선택 시** 상권·평가 등 부가 조회가 발생합니다. 자동 폴링은 두지 않는 것이 쿼터에 유리합니다. |
| **과도 호출 완화(코드 방향)** | 동일 조건 연타·디바운스·서버 캐시(Redis 등)는 **상용 단계**에서 설계. PoC는 **사용자 의도 있는 조회** 위주로 유지하는 것이 문서·심사 설명과 일치합니다. |

**Google Generative AI:** Google Cloud 콘솔·요금제에 따른 **RPM/TPM/일일 한도**가 별도입니다(`GOOGLE_GENERATIVE_AI_API_KEY`).

**네이버 지도:** NCP 콘솔의 **Maps API 사용량 정책**을 따릅니다(`NEXT_PUBLIC_NAVER_CLIENT_ID`).

## 5. 상용화 체크리스트 (요약)

1. 운영 주체·민원·장애 연락처를 내부 공문 또는 서비스 소개에 명시  
2. 공공데이터포털·NCP·Google 각각의 **쿼터·과금** 모니터링 대시보드 연결  
3. 원천 갱신 주기가 긴 지표는 UI에 **“데이터 기준일”** 표시(가능 시 API 응답 필드 활용)  
4. 서버 캐시 도입 시 **캐시 TTL**을 내부 문서에 명시

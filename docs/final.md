# 생활정보홍보우편 스마트 영업 네비게이터(PoC) — 심사·실증 요약

## 1. 제목

**공공데이터·지도·AI를 활용한 생활정보홍보우편 B2B 스마트 영업 네비게이터 실증(PoC)**  
*배포형 웹 서비스(Vercel 등)에서 상권·타겟·견적·제안 자료까지 한 흐름으로 시연*

---

## 2. 개요

본 PoC는 우체국 **생활정보홍보우편** B2B 영업 과정에서 필요한 **상권·상가·배후 수요(아파트 세대 등)** 정보를 **공공데이터 API**와 **지도** 위에 모아 보여 주고, **우편물 형태·수량·감액 조건**을 반영한 **견적 산출**, **상담·제안에 쓸 문구 초안**, **인쇄·PDF용 제안서**까지 연결하는 **웹 기반 영업 지원 도구**입니다.

심사·실증에서는 **배포된 웹사이트**를 통해 다음을 한 번에 확인할 수 있도록 구성했습니다.

- **탐색·선정:** 행정동·업종 등 조건으로 후보를 좁히고, 지도와 목록으로 타겟을 고릅니다.
- **근거 제시:** 상권 요약·지표·추세 등을 카드 형태로 묶어 “왜 이 구역·이 업종인지” 설명 가능한 수준의 브리핑을 만듭니다.
- **실무 산출물:** 견적과 제안 흐름을 화면에서 이어 **PDF/인쇄**로 보낼 수 있게 하여, 현장에서 바로 쓸 **산출물 중심** PoC로 완성도를 보입니다.
- **선택적 AI 보조:** 외부 생성형 AI를 쓰는 환경에서는 **영업 멘트·초안 보강**을 서버 경유로 요청할 수 있고, 키가 없거나 정책상 막힌 환경에서는 **기능이 비활성**되며 사용자에게 **한글 안내**로 이유를 알립니다.

또한 PoC 특성상 **실제 행정통계·실매출 원천**과 **시연·가공으로 만든 지표**가 섞일 수 있는 영역은, **화면과 PDF에 반복 고지**하여 심사자가 “무엇이 실데이터이고 무엇이 시연용인지”를 **사이트만 보고도** 구분할 수 있게 했습니다.

## 3. 현황 및 문제점

### 3.1 업무 관점의 문제(개선 동기)

기존 B2B 영업은 **후보 발굴·배후 수요 설명·비용 대비 효과 설명**에 시간이 많이 들고, **담당자 역량·자료 준비 속도**에 따라 제안 품질 편차가 큽니다. 수기로 견적과 제안서를 맞추면 **재작업·산술 오류**가 발생하기 쉽습니다.

### 3.2 PoC가 해결하려는 초점

본 PoC는 “정보 나열”이 아니라 **영업 대화와 계약 준비에 바로 연결되는 산출물**(견적 근거, 설명 카드, PDF)을 **짧은 시간 안에** 만들도록 설계했습니다.

### 3.3 PoC 운영·신뢰 측의 한계(문제점으로 솔직히 명시)

1. **데이터의 성격 혼동 위험**  
   상가 목록 등은 공공 API 기반이지만, 시연을 위해 **가공·모의 규칙**으로 만든 매출·평가류 지표가 포함될 수 있습니다. 이를 실제 원천 통계로 오인하면 제안 신뢰가 떨어지므로, **고지 없는 시연은 오히려 감점 요인**이 됩니다.

2. **“실시간”의 의미**  
   화면 수치는 사용자가 조회한 시점의 API 응답에 따릅니다. 원천 DB의 갱신 주기는 각 기관 정책이며, 앱이 통제하는 영역이 아닙니다. 따라서 **“항상 최신의 행정 확정치”**라는 식의 과장 표현은 피해야 합니다.

3. **외부 API 쿼터·과금**  
   공공데이터포털, 지도, 생성형 AI는 각각 **이용 조건·한도·과금**이 다릅니다. PoC는 **사용자가 의도적으로 조회·변경할 때** 위주로 호출되도록 설계하는 것이 일반적으로 유리하나, 트래픽이 커지면 **운영 주체의 모니터링·키 관리**가 필수입니다.

4. **내부망·보안 환경**  
   공인망 PoC와 달리, 일부 기관은 인터넷 직접 연결이나 특정 벤더 API 사용이 제한됩니다. 이 경우 AI 보조는 비활성되거나, 조직이 허용한 **게이트웨이/프록시 경로**로 전환해야 합니다(배포 사이트만으로는 “망 분리 이행 완료”를 증명하기 어렵고, **도입 단계 과제**로 남는 것이 정상입니다).

5. **운영·민원·SLA**  
   PoC는 실증용이며, 상용 서비스처럼 **민원 창구·장애 대응 SLA**까지 포함한다고 보기 어렵습니다. 확산 논의는 “기능”과 별도로 **운영 주체 확정**이 전제되어야 합니다.

## 4. 개선방안

### 4.1 심사·시연 관점: “웹에서 즉시 확인되는 완성도”

- **한 화면 흐름 완결:** 지도·필터·핵심 지표·견적·제안 출력까지 끊기지 않게 구성합니다.
- **빈 데이터·오류·로딩:** API 실패, 키 미설정, AI 비활성 등은 **사용자가 이해할 수 있는 한글 메시지**로 처리해 “데모 중 멈춤”을 줄입니다.
- **PDF·인쇄 품질:** 심사 환경은 종종 프로젝터·브라우저 창 중심이므로, **인쇄 미리보기/PDF 상단**에도 **데이터 성격·조회 시점·가공 여부**가 드러나게 하여 오해를 예방합니다.

### 4.2 신뢰·윤리: 외부 AI를 쓰는 경우의 최소 기준

- **민감 패턴 마스킹 후 전송:** 사용자가 입력한 문장에 포함될 수 있는 연락처·식별번호 형태 등을 **모델 전송 전에 치환**합니다.
- **로그 최소화:** 운영 환경에서는 **원문 프롬프트를 남기지 않는 방향**을 원칙으로 하여, 감사·유출 리스크를 낮춥니다.
- **명시적 비활성 경로:** 키가 없거나 정책상 사용 불가일 때 **조용히 실패**하지 않고, **기능 범위를 한글로 안내**합니다.

### 4.3 운영 준비(상용 전환 로드맵)

- **역할 분리:** PoC는 과제 실증 범위로 두고, 상용화 시에는 **도입 기관(우정청 등)** 이 서비스 소유·배포·민원을 담당한다는 전제를 문서·화면 고지로 정리합니다.
- **키 거버넌스:** 공공 API·지도·AI 키는 개인 PC가 아니라 **조직 발급·회전·권한**으로 이관합니다.
- **쿼터 모니터링:** 포털·클라우드 콘솔의 사용량을 주기적으로 확인하고, 필요 시 **캐시·디바운스·배치**로 호출을 줄입니다(대규모 트래픽은 PoC 이후 과제).
- **내부망 대응:** 조직이 제공하는 **허용 엔드포인트(게이트웨이)** 로 모델 호출 접두를 바꾸는 방식을 전제로 하되, **프로토콜 호환**은 네트워크 담당자 확인이 필요합니다.

### 4.4 검증(개발·배포 파이프라인 관점)

배포 사이트 체험과 별개로, PoC 안정성을 위해 **빌드 전 환경 점검(필수 키 존재·샘플 호출 등)** 을 자동화해 “데모 전날 키 누락” 같은 사고를 줄입니다. 다만 심사자 체험의 핵심은 여전히 **배포 URL에서의 시연 품질**입니다.

## 5. 기대효과

### 5.1 현장 효율

상권 근거 정리·견적·제안 자료 초안까지의 준비 시간을 줄이고, 같은 조건이라면 **산출물의 일관성**을 높여 **재작업**을 줄입니다.

### 5.2 고객(소상공인) 신뢰

공공·가공·시뮬레이션을 **화면과 PDF에서 구분**해 제시함으로써, 과장된 단정을 피하고 **설명 가능한 제안**으로 전환합니다.

### 5.3 조직·확산

데이터 기반 설명이 가능해지면 교육·온보딩 비용을 줄이고, 전국 단위로 기능을 복제할 때도 **동일한 판단 프레임**을 공유하기 쉬워집니다. 다만 확산은 **운영 주체·키·쿼터·망 정책**이 정리된 뒤가 현실적입니다.

### 5.4 심사 관점(가점·감점 요인과의 정합)

- **완성도:** 배포형으로 “보이는 산출물” 중심 평가에 대응합니다.
- **안전성·윤리:** 외부 AI 사용 시 **마스킹·로그·비활성 안내**로 리스크를 낮춥니다.
- **지속 운영:** PoC와 상용의 경계(운영 주체·한도·갱신 의미)를 명확히 하여 **과장 없는 로드맵**을 제시합니다.

## 부록: 심사 시연 체크리스트

1. **필터 변경 → 목록/지표 갱신**이 자연스러운지
2. **가공·시연 지표 고지**가 눈에 띄는지(특히 PDF 상단)
3. **AI 보조**가 켜진 환경/꺼진 환경 각각에서 **안내 문구**가 적절한지
4. **API 오류**를 넣었을 때(의도적 차단 등) **복구 메시지**가 있는지
5. **견적·제안서**가 실제 영업 대화에 쓸 수 있는 수준의 정보 밀도인지
